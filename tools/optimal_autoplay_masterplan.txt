OPTIMAL AUTOPLAY MASTER PLAN

Purpose
- This document is a guide for the current agent responsible for turning autoplay into a credible near-optimal benchmark player, exhaustive case handler, and game-audit tool.
- The goal is not only to make the current heuristic bot slightly better.
- The goal is to build a benchmark player that can answer the real design question: if a strong, informed player plays this game correctly, how often should they survive, progress, and win while engaging with the full useful game system surface?
- Seeded runs are for optimization, regression protection, and replaying specific failure classes.
- Random runs are for statistics: how strong the benchmark player is against the game as it actually exists.

What this document contains
- A concrete diagnosis of why the current autoplay underplays the game.
- A target architecture for a maintainable, near-optimal autoplay system.
- A specific implementation plan tied to the current codebase.
- A benchmark rubric so future versions can be judged against a stable standard.
- Maintenance guidance so the system stays useful as the game evolves.
- A practical checklist for expanding coverage until the harness can engage every meaningful winning path and expose fairness problems or broken systems when it cannot.

CORE GAME MECHANIC TRUTHS (verified — do not work against these)
======================================================================

NO ON-FOOT TRAVEL
- Without a car there is NO afternoon activity whatsoever.
- make_shop_list(on_foot=True) always returns an empty list and the game
  immediately calls night_event().  The Doctor, Convenience Store, Pawn Shop,
  Marvin, Loan Shark — none are reachable on foot.
- Any routing logic conditioned on "no car AND in the afternoon menu" is dead
  code.  Mechanics reach the player as random day events; the player cannot
  seek them out on foot.

MECHANIC SHOPS UNLOCK ONLY VIA SUCCESSFUL CAR PURCHASE
- Meeting Tom / Frank / Oswald during their roadside intro event does NOT
  unlock their shop.  Only paying for and receiving the car from that mechanic
  unlocks their afternoon shop listing.

MECHANIC INTRO SYMPATHY MECHANICS (verified from mechanics_intro.py)
- Tom  (price $150-$350): say YES even when broke → 50 % chance Tom offers a
  $50 discount.  No money is taken if still unaffordable.  Always say yes.
- Frank (price $50-$100): say YES when broke → adds a "Frank" Danger.  BAD.
  Always say NO to Frank when balance < quoted price.  40 % success rate even
  when paid (Frank is unreliable).
- Oswald (price $800-$900): say YES when CAN afford → tip $50-100 AND car
  fixed.  Say YES when broke → tip $50-100, no fix.  Say NO → no tip, just
  leaves.  Always say yes to Oswald (tip even if broke).

DOCTOR PRICING (verified from locations.py visit_doctor)
- No insurance:     35–60 % of current balance
- Faulty Insurance: 10–35 % of current balance (10 % ban risk)
- Real Insurance:   FREE
  (The docs say 65–90 % — the code is the ground truth: 35–60 %.)

MARVIN AVAILABILITY (verified from lists.py make_shop_list)
- Marvin appears in the shop list whenever the player has a Map or Worn Map.
  There is NO rank gate in make_shop_list — only the map requirement.
  The docs say Rank 3+ but the code has no rank check here.
- Marvin's item prices are RANDOM per visit (not fixed):
  e.g. Health Indicator $8,000–$9,500, Golden Watch $29,000–$35,000,
  Faulty Insurance $10,000–$12,000.  Cheapest item starts at $8,000.
  All items use random.choice([low, mid, high]).

ROADSIDE MECHANIC VISIT (newly implemented — not in docs, added for longevity)
- Fires at 45 % probability when car trouble adds "Wasted Afternoon".
- Only available once the player has bought a car from that mechanic
  (has_met("Tom"), has_met("Frank"), or has_met("Oswald")).
- Randomly picks from all eligible mechanics the player knows.
- Prices: Tom $100–$200, Frank $75–$150, Oswald $150–$250.
- Tom roadside: same sympathy as intro (50 % discount offer if broke).
- Frank roadside: 40 % success rate; adds Frank Danger if YES + broke.
- Oswald roadside: tip $50–$100 on YES (both afford and broke); no tip on NO.

MECHANIC INTRO FIRING ORDER (all are random DAY EVENTS, not menu choices)
- All three mechanics have EQUAL probability of appearing — the order is
  random, not fixed.  Each is picked from a pool of eligible, not-yet-met
  mechanics at the moment the storyline event fires.
- Tom  eligible when: balance >= 200, Tom Event not yet met (no car check).
- Frank eligible when: NO CAR ONLY, balance >= 200, Frank Event not yet met.
- Oswald eligible when: balance >= 850, Oswald Event not yet met (no car check —
  Oswald can visit the player both early and late in the run).
- If you truly lose your car (no item in inventory), you cannot get it back.
  The mechanic intro events only *give* a car — they do not recover a lost one.

MARVIN PRICING (random ranges, set fresh each visit)
- Cheapest items start at $8,000.  Visit ONLY when marvin_affordable_priority
  > 0 (bot can afford at least one item at its minimum price estimate).
  Scouting visits when nothing is affordable waste an afternoon with no gain.

WITCH DOCTOR PRICING
- Healing costs 5–25 % of current balance (random).
- Potions are randomly priced per visit from fixed ranges (e.g. No Bust
  $25k–$30k, Fortunate Day $12k–$18k).
  Only visit when balance can absorb both the heal cost and a target potion.

LOAN SHARK (VINNIE)
- Vinnie is first met via the "vinnie_referral_card" day event, which fires
  only when the player already has a car.  The menu entry does not exist
  until after that event.

Autoplay philosophy
- Autoplay is not only a bot-quality project. It is also a gameplay audit instrument.
- The benchmark player should try to engage every mechanic that can reasonably improve a run: car progression, doctor care, witch doctor, Marvin, pawn shop, loan shark, companions, gifts, convenience-store synergies, crafting, workbench loops, and endings.
- The harness should prefer the best available outcome in events, story branches, and shop decisions unless status pressure makes a different choice strategically necessary.
- Status-prioritized decisions are part of optimal play:
  - survival and emergency stabilization outrank greed
  - interrupted or locked-out days must be recorded explicitly
  - debt, sanity collapse, fatigue, companion upkeep, and durability churn are planner responsibilities
- If the benchmark player is not engaging with a major mechanic, that is a useful finding. The likely causes are:
  - the autoplay does not understand the system yet
  - the unlock path is too weak or too hidden
  - the mechanic is bugged, absent from real gameplay, or underpowered relative to its setup cost
- If repeated harness evidence shows a branch or event is unfair, too punitive, too repetitive, or too weak to support progression, the correct response may be a game-balance fix rather than more autoplay heuristics.
- The harness should be used to compare expected mechanics against actual gameplay. If a system should appear in real runs and does not, treat that as a bug-finding use case and inspect the core game code.

Current harness expansion priorities
- The current harness already covers core blackjack decisions, route selection, medical interruption behavior, mechanic thresholds, and some Marvin preservation behavior.
- The biggest remaining coverage gaps are not generic "more heuristics". They are incomplete optimal-play surfaces with real long-horizon value:
  - gifts and gift wrapping timing
  - dealer happiness thresholds and gift-system effects actually appearing in real runs
  - companion acquisition, hunger, happiness, runaway prevention, and companion-specific route pressure
  - convenience-store synergies by rank and by current build state
  - crafted-item loops, workbench usage, and downstream event value from crafted items
  - pawn-shop value chains beyond one-off collectible liquidation
- The immediate autoplay priority is full coverage and verification of the mechanics already present so the benchmark can engage every meaningful winning path before new content breadth is added.
- Content expansion ideas such as more store items, more item-granting events, more crafting outputs, and more pawnable or craft-reactive events are valid game-design follow-ons, but they should not replace coverage and verification of the systems that already exist.
- If current systems are not giving the player enough positive push to grow stronger over time, the harness should surface that clearly so the game can be rebalanced through event weighting, reward strength, repeatability controls, or branch tuning.

Core judgment
- The current autoplay in tools/quicktest.py is useful scaffolding and a decent fuzz-style regression harness.
- It is not a credible benchmark for strong play.
- Its losses currently reflect both game difficulty and bot weakness, but bot weakness is the larger issue.
- The main failure is still architectural, but the practical mission is broader than architecture alone: the benchmark must cover the full useful game surface, find missing or unfair systems, and explain why major winning routes are not being realized.

PART 1: CURRENT SYSTEM REVIEW
======================================================================

Relevant files
- tools/autotest.py
- tools/quicktest.py
- blackjack.py
- story/game_flow.py
- story/locations.py
- story/storylines.py
- story/player_core.py
- story/endings.py
- story/events_illness.py
- lists.py

What the current system does well
- tools/autotest.py is a competent batch runner and reporting layer.
- The current harness captures a lot of useful telemetry: survival, deaths, unlocks, visits, item provenance, fallback decisions, doctor misses, and aggregate reports.
- The current autoplay already understands more than a trivial bot does:
  - blackjack actions such as split, surrender, double, insurance, peek, replay
  - route choices such as doctor, witch, store, pawn shop, loan shark, Marvin, mechanics, workbench, adventure, stay home
  - some reserve logic around doctor costs, car needs, rank thresholds, and store budgets

What the current system fundamentally is
- tools/quicktest.py is still a very large policy blob, but it is no longer only prompt scraping.
- It monkey-patches blackjack runtime methods and still relies too much on RECENT_TEXT and menu text in some places.
- It also already contains useful ingredients that should be preserved: state extraction, strategic goal concepts, scenario cases, route tracing, and machine-readable policy surfaces under tools/autoplay/.
- The practical problem is not that nothing exists. The problem is that coverage is still incomplete, long-horizon optimization is still uneven, and too many important mechanics are not being deliberately engaged often enough.

Why the current autoplay is not a strong benchmark

1. It is prompt-driven instead of state-driven
- The bot relies heavily on RECENT_TEXT, menu scraping, prompt fragments, string matching, and handcrafted overrides.
- Examples:
  - _get_recent_menu_options
  - _event_yes_no_override
  - _event_option_override
  - _detect_mechanic_name
  - _looks_like_mechanic_intro_offer
- This makes it brittle and encourages reactive behavior rather than reasoning over game state.

2. It has no explicit global objective function
- There is no single state value function or long-horizon objective being optimized.
- Instead, decisions are distributed across local heuristics like:
  - _wants_store_run
  - _wants_marvin_run
  - _wants_mechanic_progression_run
  - _wants_upgrade_run
  - _wants_millionaire_push
  - _should_visit_doctor
  - _auto_bet
- These functions are individually plausible, but they do not compose into a coherent strategy.

3. It has no explicit progression planner
- The real game has a progression graph with prerequisites.
- Major routes are gated by rank, day, balance, item ownership, breadcrumb events, health/sanity thresholds, mechanic arcs, and special unlocks.
- The bot does not model that graph directly.
- Instead it chooses destinations from a rotating or priority-ordered list in _choose_progression_destination and _choose_destination.
- This is not planning. It is weighted reaction.

3A. It does not yet cover the full useful mechanic surface reliably enough
- Optimal play in this game is not just about betting and routing.
- The benchmark must deliberately use and validate:
  - gifts and gift-wrapping systems
  - dealer happiness effects
  - companions and companion upkeep
  - convenience-store build synergies by rank and run state
  - pawn-shop realization loops
  - crafting and workbench loops
  - millionaire branches and endings
- If these systems do not appear in real runs, that is not background noise. It is a direct benchmark finding.

4. Blackjack decisions are simplified and locally adjusted
- _recommended_blackjack_action is basic-strategy-like and hand-authored.
- _auto_hit_or_stand adds peek-based local correction.
- _should_take_insurance only buys insurance when Dealer's Whispers confirms dealer blackjack.
- _should_replay_hand uses reserve and loss-ratio heuristics.
- _auto_bet aggressively targets rank thresholds, but does not optimize expected run value under health/progression risk.
- This is smarter than a naive bot, but still far from a game-specific EV policy.

5. Many thresholds are magic numbers without a stable policy framework
- The file has a large number of hardcoded thresholds:
  - health/sanity cutoffs
  - balance gates
  - item priority weights
  - reserve targets
  - visit cooldowns
  - rank-specific bet ratios
  - loan triggers
  - stall-day conditions
- The problem is not that thresholds exist.
- The problem is that they are distributed across many functions without a formal policy layer, explanation layer, or validation layer.

6. The bot mixes four different problem types into one file
- The current file tries to handle all of this in one place:
  - IO adaptation and prompt interception
  - event/menu parsing
  - strategic planning
  - tactical route choice
  - blackjack action choice
  - economy and reserve management
  - reporting side-effects
- This makes it hard to improve one area without breaking another.

7. The current system underuses the actual state richness already exposed by the game
- The game code already has real state that can be reasoned over directly:
  - health, sanity, status effects, injuries
  - debt, fraudulent cash, dealer happiness
  - rank, balance, day
  - companions
  - chosen mechanic, mechanic visits, broken items, repairing items
  - map ownership, car ownership, unlock conditions, progression visits
- The bot still behaves as if text prompts are the primary truth source.

High-value code observations from the current implementation

Destination selection
- _choose_progression_destination is the main progression router.
- It assembles possible route choices, then picks from ordered lists based on rank/day/rotation.
- This is a key weakness: the route ordering is a handcrafted priority stack, not the result of explicit goal selection.
- _choose_destination then overlays medical urgency, recovery days, pawn cashouts, store runs, and stay-home fallbacks.
- Result: sensible local choices, weak long-horizon coordination.

Blackjack controller
- _recommended_blackjack_action provides simplified action selection.
- _should_use_peek and _auto_hit_or_stand refine the action with a one-card lookahead.
- _auto_bet tries to push aggressively toward rank thresholds with safety checks.
- Result: decent tactical behavior, weak run-level optimization.

Medical policy
- _needs_doctor, _doctor_visit_is_urgent, _doctor_need_score, _wants_witch_heal, and _should_visit_doctor are among the best parts of the current bot.
- They are still threshold-based, but they at least attempt seriousness grading.
- The main weakness is that they are not fed by a larger survival planner or replay suite of saveable-death scenarios.

Progression stalling logic
- _progress_stall_days tracks lack of progress by rank/balance/car/Marvin access.
- This is useful telemetry and should be preserved.
- It should become an input to a real planning system, not just another local heuristic trigger.

Conclusion on current system
- Keep the reporting and telemetry ideas.
- Keep some of the current state feature extraction.
- Keep the validated scenario suites, state modeling, and planner-oriented modules already extracted.
- Expand coverage until every major mechanic that should help a winning run is either deliberately used or explicitly explained as blocked, bugged, unfair, or underpowered.
- Replace the decision architecture where needed, but do it in a way that preserves current guardrails and fixed-seed learning.

PART 2: TARGET AUTOPLAY ARCHITECTURE
======================================================================

Design goal
- Build a benchmark autoplay system that behaves like a strong, informed human with stable strategic priorities and good tactical blackjack play.
- The system must be maintainable, inspectable, robust against writing changes, and suitable for auditing whether the game itself is fair and fully functioning.

Non-goals
- Do not keep expanding the current heuristic monolith.
- Do not treat prompt parsing as the long-term interface.
- Do not optimize only for one-night bankroll growth if that degrades overall win rate.
- Do not confuse missing mechanic engagement with proof that the mechanic is bad before checking whether the mechanic actually appears and functions in gameplay.

Architecture overview
- Split autoplay into five layers.

Layer 1: Adapter layer
Purpose:
- Interface with the existing game runtime.
- Capture structured state.
- Translate game prompts into stable decision requests.

Responsibilities:
- Hook or wrap menu prompts and yes/no prompts.
- Build structured choice objects rather than relying on raw prose where possible.
- Extract game state snapshots before and after decisions.
- Maintain compatibility with current runtime until deeper refactors are made.

Outputs:
- GameState snapshot
- DecisionRequest object
- DecisionTrace object

Layer 2: State model layer
Purpose:
- Normalize the real game into a machine-friendly strategic state.

State categories to model explicitly:
- Core:
  day, balance, rank, health, sanity, alive, location context
- Risk:
  statuses, injuries, urgent medical flag, doctor affordability, witch affordability
- Progression:
  car owned, map access, Marvin access, Witch met, mechanic route availability, mechanic visits, upgrade access, millionaire status, ending opportunity
- Economy:
  loan debt, warning level, fraudulent cash, pawn sale potential, store inventory opportunities
- Combat/gambling edge:
  active blackjack items, flask effects, broken items, repairing items, dealer happiness, extra-round items
- Resources:
  inventory, flasks, companions, food, repair kits, emergency items
- Opportunity flags:
  can unlock car, can advance mechanic arc, can visit Marvin profitably, can visit witch profitably, can adventure safely, can convert millionaire state to ending

Critical state dynamics the planner must model explicitly
- The real run is not just route choice plus blackjack.
- Several systems can pre-empt normal planning or change what decisions even exist.

Interrupt hierarchy
- The planner must recognize that many afternoons never reach normal destination choice.
- A practical decision-order model should look like this:
  - daily passive updates and forced morning systems
  - millionaire morning override if active
  - storyline event override if active
  - fatigue-based day-event suppression
  - afternoon travel restriction lockouts such as Wind, Rain, Battery, Engine, or Wasted Afternoon
  - random car trouble interruption
  - companion-afternoon diversion if offered and chosen
  - only then normal destination selection
- The autoplay should expose this ordering in traces so a missed route is not misdiagnosed as a bad route policy when the route was never actually available.

Recurring resource churn cycles
- The benchmark player must manage recurring drains, not just static reserves.
- Important cycles include:
  - fatigue accumulation and recovery
  - sanity loss, broken-state pressure, and madness confrontation risk
  - companion hunger, happiness decay, bonding, runaway risk, and companion-specific food pressure
  - debt growth through interest and warning escalation
  - fraudulent cash accumulation and blending through blackjack
  - item durability loss, breakage, repair queues, and repair opportunity cost
  - flask acquisition, active effects, expiry, and replacement timing
- These cycles should be visible in state snapshots as first-class planner inputs, not hidden as local tactical heuristics.

High-consequence branch systems
- Some choices are not just local EV decisions; they permanently redirect the run.
- The planner and trace layer should account for:
  - millionaire conversion branches such as mechanic ending route, airport escape, or continued gambling
  - sanity collapse branches such as full madness ending versus surviving as Broken
  - storyline yes/no branches with run-ending or run-shaping outcomes
  - companion-heavy secret ending routes and betrayal-style dark branches
- These should be modeled as branch funnels with explicit prerequisites, not treated as generic event choices.

Layer 3: Strategic planner
Purpose:
- Choose the current top-level run goal.

This is the missing core of the current bot.

Planner output should be one explicit StrategicGoal at a time, such as:
- survive_emergency
- stabilize_health
- acquire_car
- preserve_car
- bootstrap_blackjack_edge
- unlock_marvin
- exploit_marvin
- advance_mechanic_arc
- repair_or_upgrade_gear
- reduce_debt_risk
- push_next_rank
- reach_adventure_threshold
- exploit_adventure
- convert_millionaire_to_ending

Planner requirements:
- Must use explicit prerequisites and blockers.
- Must score long-horizon expected value, not just immediate convenience.
- Must explain its choice in a trace.
- Must distinguish between planned actions and interrupt-driven fallback handling.

Strategic systems that deserve dedicated planner support
- Debt is not just a reserve penalty.
- The loan shark system is a full strategic subsystem with:
  - borrow versus avoid-borrow policy
  - repay timing policy
  - warning-tier escalation handling
  - forced-encounter risk management
  - fraudulent-cash blending strategy
- Sanity is not just another hit point track.
- The planner should explicitly track low-sanity danger tiers, broken-state tolerance, and whether a run is still pursuing stable survival or is already in a degraded high-variance branch.
- Companion management should also be strategic, not only tactical.
- Companion count, happiness, feeding pressure, and secret-ending potential all affect whether the run should pursue growth, maintain the current roster, or reduce obligations.

Example planner reasoning format:
- current_goal = acquire_car
- blockers = insufficient_cash, urgent_health_risk
- supporting_actions = doctor_visit, safe_betting, pawn_sale
- suppressed_actions = Marvin, store_luxury, adventure

Layer 4: Tactical policy layer
Purpose:
- Given a strategic goal, choose the best immediate action.

Sub-policies:
- route_policy
- blackjack_policy
- purchase_policy
- medical_policy
- debt_policy
- repair_policy
- event_choice_policy

Each sub-policy should accept:
- GameState
- StrategicGoal
- DecisionRequest

And return:
- chosen_action
- reason
- confidence
- expected_value_estimate or utility_estimate

Layer 5: Evaluation and reporting layer
Purpose:
- Run seeded benchmarks.
- Store machine-readable traces.
- Report strategic funnels and failure categories.
- Report mechanic presence, fairness candidates, and expected-but-missing systems.

This layer can remain close to tools/autotest.py, but should prefer structured outputs over regexing plain text.

Step-by-step implementation plan

Visible-mechanics checklist
- The benchmark is not complete until the following systems visibly appear in real runs often enough to evaluate them:
  - car acquisition and mechanic progression
  - doctor and witch-doctor stabilization paths
  - worn-map and map unlock paths
  - Marvin visits and Marvin purchase-to-use funnels
  - convenience-store purchases by rank and build context
  - pawn-shop liquidation and follow-up value realization
  - loan shark borrowing, repayment, warning escalation, and fraudulent-cash blending
  - gifts, dealer happiness thresholds, and gift effect payoffs
  - companions, feeding, happiness, runaways, and companion-positive branches
  - crafting inputs, workbench visits, outputs, and downstream item-event payoffs
  - adventure unlocks and adventure exploitation
  - millionaire reach, millionaire branch menus, airport route, mechanic route, and ending conversion
- If a listed system does not appear, the next agent should explicitly decide whether the cause is:
  - autoplay incompleteness
  - progression weakness or balance weakness
  - missing content surface in normal gameplay
  - a core game bug

Multi-process iteration plan to 1 million dollars
- The implementation should proceed as a deliberate multi-process loop, not a single rewrite.
- Process 1: Coverage process
  - add scenario cases and traces until every major route, event family, and subsystem has explicit coverage
  - verify that mechanics expected in normal gameplay actually appear in runs
- Process 2: Prevention process
  - use fixed seeded failures to identify the most common preventable deaths, stalls, and underpowered branches
  - fix the most common preventions first
- Process 3: Fairness and balance process
  - when repeated evidence shows an event or branch is too punitive, too repetitive, or too weak to support progression, tune the core game rather than hiding the issue with heuristics
- Process 4: Progression process
  - deliberately improve the run funnel in stages:
    - get off your feet consistently
    - stabilize early survival and medical handling
    - convert car access into rank 1 and map access
    - convert map access into reliable 8k to 10k bankroll growth
    - convert 10k+ runs into Marvin leverage and stronger blackjack edge
    - convert strong late-game runs into 100k, then 500k, then 1 million dollars
    - convert millionaire state into endings deliberately rather than incidentally
- Process 5: Statistics process
  - use random seeded batches to measure whether the benchmark is actually getting stronger against the full game
  - use fixed-seed suites to protect known wins and prevent regressions
- A run to 1 million dollars should not be treated as a lucky outlier. The benchmark should eventually make that path deliberate, explainable, and repeatable enough to evaluate the game around it.

Phase 5: Replace blackjack policy with EV-based evaluation
Current weak points to replace:
- _recommended_blackjack_action
- _should_use_peek
- _should_take_insurance
- _should_replay_hand
- _auto_bet

Desired blackjack policy behavior
- Compute expected value of hit, stand, double, split, surrender, insurance, peek, replay under the custom rules.
- Include custom items and flask effects.
- Include durability costs and opportunity costs.
- Include run-level reserve constraints.

Two practical implementation options

Option A: Exact or semi-exact evaluator
- Best if deck state is available enough to simulate accurately.
- Build outcome trees for current hand and dealer upcard.
- Harder due to custom mechanics and special effects.

Option B: Monte Carlo evaluator
- Recommended first.
- For the current blackjack state:
  - clone the hand/dealer/deck state if possible
  - simulate many rollouts for each action
  - estimate expected delta in bankroll and strategic value
- Add a run-value term, not just immediate chip EV.

Blackjack action utility should combine:
- immediate expected chip value
- probability of bust/loss/push/win/blackjack
- reserve preservation value
- rank-threshold progress value
- durability consumption cost
- effect consumption cost

Bet sizing policy redesign
- Do not let betting be only a rank-threshold pressure function.
- Bet sizing should consider:
  - current goal
  - bankroll volatility tolerance
  - survival reserve
  - doctor/witch needs
  - pending route opportunities
  - current blackjack edge from items/flasks
  - debt pressure
  - millionaire conversion strategy

Good design pattern:
- target_risk_profile = conservative, balanced, aggressive, conversion_push
- choose profile from strategic goal
- use profile to set bet bands

PART 5: MAINTENANCE RULES
======================================================================

Rule 1: Prefer structured state over text scraping
- If a decision can be made from player/game state, do not make it from RECENT_TEXT.
- Raw text is a compatibility fallback only.

Rule 2: Every major policy decision must be explainable
- If a route or blackjack action cannot emit a reason trace, the policy is not maintainable.

Rule 3: Add scenario tests when a death or failure looks avoidable
- Every obvious avoidable death should become a replayable case.
- The system should learn from specific failure classes.

Rule 4: Keep strategy and IO separated
- Prompt parsing, monkey-patching, and runtime adaptation should not contain strategic reasoning.
- The strategic layer should consume normalized DecisionRequest and GameState objects.

Rule 5: Centralize tunable weights
- Item priorities, goal weights, reserve rules, aggression profiles, and route weights should live in policy config tables, not scattered magic numbers.

Rule 6: Optimize for run value, not only immediate chips
- A winning benchmark player in this game is not just a good blackjack hand player.
- It is a good run manager.

Rule 6A: Make interrupts and lockouts visible in traces
- If a day was consumed by fatigue, weather, car trouble, debt violence, companion care, or a storyline override, the trace should say so explicitly.
- Otherwise route-policy regressions and content-balance problems will be misread.

Rule 6B: Treat recurring drains as planner responsibilities
- Fatigue, sanity, debt, durability, fraudulent cash, and companion upkeep should not be scattered as local reserve checks.
- They belong in the strategic layer and should appear in planner scoring and benchmark reporting.

Rule 7: Protect reporting continuity during migration
- Preserve enough of the current tools/autotest.py outputs that historical comparison remains possible.
- Add JSON outputs instead of replacing text immediately.

Rule 8: Do not claim optimality too early
- Use language like benchmark_player, strong_policy, or near_optimal_candidate until the metrics support stronger claims.

======================================================================
PART 7: PRACTICAL PITFALLS TO AVOID
======================================================================

Pitfall 1: Trying to make the current quicktest file clean by adding more helpers
- That will improve readability somewhat but not solve the planning problem.

Pitfall 2: Replacing route heuristics before building a strategic goal layer
- Route scoring without a goal will just become a different heuristic soup.

Pitfall 3: Overfitting to a small seed set
- A few good seeds can make a bad policy look smart.

Pitfall 4: Evaluating blackjack only by immediate chip EV
- In this game, cash reserve and progression opportunities matter.

Pitfall 4A: Treating forced systems as route-policy failures
- A missed doctor, Marvin, or mechanic visit may be caused by fatigue suppression, travel restriction, car trouble, or debt interruption rather than poor destination scoring.
- If the trace does not distinguish this, tuning will target the wrong layer.

Pitfall 4B: Modeling items and companions as static assets
- Edge items break.
- Flasks expire.
- Companions require upkeep and can leave.
- A planner that assumes monotonic growth in power will overestimate run stability.

Pitfall 5: Ignoring maintainability telemetry
- If fallback prompt handling grows again, the architecture is drifting backward.

Pitfall 6: Calling the new bot optimal before it can convert major systems reliably
- It needs to show stable exploitation of Marvin, Witch, mechanics, adventures, and endings before it can be trusted as a balance benchmark.
