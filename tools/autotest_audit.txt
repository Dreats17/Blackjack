Executive assessment
- tools/autotest.py is a solid batch runner and reporting layer.
- tools/quicktest.py is the real autoplay policy engine.
- The real game is materially more complex than the harness alone suggests: blackjack EV is only one part of the problem.
- As an observability tool, this setup is good.
- As an "optimal player," it is not close to optimal yet.

Bottom line
- The harness measures a lot of useful things, but the policy it is measuring is still mostly heuristic, brittle, and heavily coupled to prompt text.
- The actual game also has hard progression gates around Marvin, Witch, mechanics, adventures, airport endings, debt, and medical status handling.
- Because of that, some weak metrics reflect both policy quality and real game structure.
- The current autoplay appears better at broad exploration and regression spotting than at maximizing survival, bankroll growth, or ending progression.

Interpretation

- Marvin and Witch are also not simple menu entries. story/storylines.py requires a Car plus specific breadcrumb events before those routes exist.
- The system is instrumented well enough to show these problems clearly, which is a strength.

Game systems the current autoplay must actually optimize across
- blackjack.py: blackjack choices plus custom actions and item effects such as insurance, peek, double, split, surrender, No Bust, Second Chance, Dealer's Hesitation, Imminent Blackjack, dealer happiness, and forced-bet pressure.
- story/locations.py and lists.py: dynamic afternoon route generation including doctor, witch, mechanics, convenience store, Marvin, pawn shop, loan shark, phone calls, workbench, and airport.

- story/player_core.py: debt, fraudulent cash, dealer happiness, pawn reputation, visit counters, companions, and gift systems.
- story/events_illness.py: much larger illness and injury surface than the original harness review implied, which increases the importance of medical triage quality.
- story/endings.py: millionaire branching and airport/mechanic endings, which means a truly strong bot should be able to convert late-game bankroll into explicit win-state progress.

Senior-engineer findings

1. High: The architecture does not match the name "autotest"
What I see:
- tools/autotest.py mostly launches tools/quicktest.py, parses text output, and aggregates metrics.
- It does not contain the actual decision policy.

Why this matters:
- Anyone reading the code can easily assume autotest is the player.
- That makes debugging slower and makes responsibility boundaries blurry.

Actionable fix:
- Rename the roles in code and docs.
- Treat tools/autotest.py as batch_runner/reporter.
- Treat tools/quicktest.py as autoplay_policy/simulator_adapter.
- Put a short module docstring at the top of both files stating that contract explicitly.

2. High: The autoplay policy is not empirically optimal by the metrics that matter
What I see:
- Large batches still show low survival and weak endgame reach.
- 300-run sample has 0 adventure visits and 0 wins.
- Unlock conversion is poor: marvin_access=120/300 but marvin=8/300.
- loan unlocks exist but actual visits are almost nonexistent.
- Some of the zeroes are partly explained by real gates. For example, adventures require rank thresholds plus prerequisite event chains, and Marvin/Witch require Car plus breadcrumbs.

Why this matters:
- A policy that unlocks content but cannot capitalize on it is not optimizing game outcomes.
- The current heuristics appear to overfit local choices and underperform on long-horizon progression.
- The correct standard is not "did the bot instantly reach hidden content" but "once prerequisites are plausibly reachable, does the bot drive toward them deliberately?" Right now the answer is still mostly no.

Actionable fix:
- Define an explicit objective score for autoplay.
- Example weighted score:
  score = survival + rank progression + peak bankroll + ending reach + content conversion - doctor_missed - dead-end loops
- Fail CI or flag regressions when core metrics decline over fixed seed sets.
- Track funnel metrics for each progression path:
  unlocked -> visited -> purchased -> used -> survived after use
- Separate funnel reporting into two stages:
  prereq_reached -> route_unlocked -> route_visited -> value_realized

3. High: Decision-making is too dependent on raw prompt text and recent output buffers
What I see:
- tools/quicktest.py makes many choices by scanning RECENT_TEXT, parsing numbered menus from output, and matching phrases like "whaddya say", "you game", "come again", and similar strings.
- Fallback Review in reports confirms that generic fallback handlers are used regularly.

Why this matters:
- This is brittle.
- Small writing changes in story text can silently change behavior.
- It also mixes UI text with game semantics, which makes policy logic fragile and hard to reason about.

Actionable fix:
- Add structured choice APIs in the game layer.
- Every prompt that matters should expose a stable decision id and structured option ids.
- Example:
  prompt_id = mechanic_intro_tom
  options = [accept_repair, decline_repair]
- Make quicktest choose on ids, not prose.
- Keep the text parser only as a temporary compatibility layer, not the primary control surface.

4. High: The betting and routing logic is a large heuristic monolith with many magic numbers
What I see:
- tools/quicktest.py contains a very large amount of threshold-driven behavior around betting, medical decisions, car repair reserves, route selection, store spending, and unlock timing.
- There are many intertwined conditions and repeated cap/reserve adjustments.

Why this matters:
- This is hard to verify and easy to regress.
- Local fixes will keep shifting behavior in unexpected ways.
- It is also difficult to answer basic questions like "why did the bot make this bet on day 17?"

Actionable fix:
- Split policy into modules:
  blackjack_policy
  bankroll_policy
  health_policy
  travel_policy
  purchase_policy
  fallback_policy
- Move tunable numbers into named policy tables or dataclasses.
- Emit a per-decision trace reason, not just final outcome summaries.
- Add golden-seed regression tests that assert policy decisions for known states.

5. Medium-High: The blackjack strategy is better than basic scripting, but still not truly optimal
What I see:
- The actual blackjack engine supports split, surrender, double, peek, insurance, and Second Chance.
- It also includes materially game-changing effects such as No Bust, Imminent Blackjack, Dealer's Hesitation, dealer happiness interactions, Lucky Coin style reversals, free-hand effects, and durability consumption.
- That is a good improvement over a naive bot.
- However, it still uses a simplified rule table and hand-authored logic rather than a game-specific EV engine.

Why this matters:
- "Optimal" in this game is not just standard blackjack basic strategy.
- The game has custom items, flasks, durability costs, insurance variants, fake cash, dealer-state effects, and progression pressure.
- A static rules table will miss EV shifts introduced by those systems.

Actionable fix:
- Build a game-specific blackjack evaluator.
- For a given state, estimate EV for hit, stand, double, split, surrender, insurance, peek, and replay.
- Include item effects, durability cost, free-hand effects, and reserve pressure.
- If exact calculation is too expensive, use Monte Carlo rollouts from the current hand state.

6. Medium: Postmortem medical analysis exists, but the learning loop is incomplete
What I see:
- tools/autotest.py reports doctor_missed and has a useful doctor_likely_saveable heuristic.
- The 300-run report shows doctor_missed=78.
- The real game now has a much wider illness/injury space in story/events_illness.py than the harness-level review alone revealed.
- That means triage quality is even more central to optimal play than the original audit suggested.

Why this matters:
- This is useful telemetry, but it is still mostly retrospective.
- The policy is not converting enough of that evidence into prevention.

Actionable fix:
- Promote postmortem categories into explicit policy test cases.
- Build a replay suite of saveable deaths.
- For each one, assert that the policy now routes to doctor or witch instead of gambling again.
- Track "saveable death rate" as a top-line metric.

7. Medium: Unlock-to-visit conversion is weak, which suggests strategy dead zones
What I see:
- Large mismatch between access and actual exploitation.
- Example from the 300-run report:
  marvin_access=120, marvin=8
  loan unlocks are much higher than visits
  upgrade and adventure usage are near zero
- The actual route surface is also deeper than a generic "shop visit" label suggests. Marvin, pawn, loan shark, workbench, airport, and phone calls all have multi-step value chains after unlock.

Why this matters:
- The bot is paying the cost to unlock systems, but not reliably harvesting the reward.
- That usually means destination policy is not stateful enough after unlock.

Actionable fix:
- Add per-system conversion dashboards:
  map acquired -> marvin visited -> item bought -> item used -> survival delta
  witch met -> witch visited -> flask bought -> flask consumed -> bankroll delta
  car acquired -> mechanic progression -> upgrade purchased -> route expansion
- Expand this to late-game systems too:
  millionaire state -> airport/mechanic branch offered -> ending selected
  tool kit acquired -> workbench visited -> part crafted/disassembled -> car reliability delta
- Use these funnels to target the biggest drop-off.

11. Medium: The harness does not model the full progression graph explicitly enough
What I see:
- Route choice in tools/quicktest.py is mostly heuristic and text-driven, but the real game has a concrete progression graph.
- make_shop_list in lists.py controls which afternoon destinations even exist.
- story/storylines.py and story/game_flow.py define hard prerequisites for major systems and endings.

Why this matters:
- Without an explicit model of prerequisites, the bot cannot reason cleanly about when it should save for a Car, push mechanic progression, chase map breadcrumbs, preserve health for rank unlocks, or convert a millionaire state into an ending.
- The result is a bot that reacts to prompts rather than planning around the actual state machine.

Actionable fix:
- Add an explicit progression planner layer.
- Represent major routes as stateful goals with prerequisites, costs, and expected payoff.
- Example goals:
  survive_to_next_day
  acquire_car
  unlock_marvin
  unlock_witch
  advance_mechanic_arc
  reach_adventure_rank_gate
  convert_millionaire_state_to_ending
- Make routing and spending decisions subordinate to the current goal, instead of relying on flat heuristics alone.

8. Medium: Report parsing is tightly coupled to human-readable text output
What I see:
- tools/autotest.py parses tools/test_out.txt with regexes against display text.

Why this matters:
- Formatting changes can break analytics even when gameplay is still correct.
- Text logs are good for humans, but weak as the primary machine interface.

Actionable fix:
- Emit structured JSON alongside the existing text report.
- Keep the text report for readability.
- Have autotest consume JSON first and fall back to text only for backward compatibility.

9. Medium: tools/test_gameplay.py is misleading and should not be trusted as evidence of optimal play
What I see:
- It says it simulates optimal gameplay, but the core round body does not actually drive a real blackjack round.
- It logs before/after balance and comments that the round logic "would need to be adapted based on actual game interface."

Why this matters:
- This file overclaims what it tests.
- It will mislead future maintainers and can create false confidence.

Actionable fix:
- Either delete it, rename it to prototype_gameplay_stub.py, or wire it to the real quicktest/autoplay hooks.
- Do not describe it as optimal gameplay until it is actually exercising the real game loop.

10. Low-Medium: Global monkey-patching and shared globals make the harness harder to evolve safely
What I see:
- tools/quicktest.py monkey-patches blackjack runtime methods and relies on globals like CURRENT_PLAYER, RECENT_TEXT, LAST_INPUT_FINGERPRINT, and counters.

Why this matters:
- This creates hidden coupling and makes parallelism, isolated testing, and reuse harder.
- It also increases the odds of state leakage between scenarios.

Actionable fix:
- Move runtime behavior behind a test adapter object.
- Inject policy and IO adapters instead of mutating module-level state where possible.
- Wrap each run in a stateful class instance instead of process-wide globals.

What this system already does well
- Good batch reporting and seed-based reproducibility.
- Strong coverage-oriented telemetry for shops, mechanics, items, event polarity, and medical cohorts.
- Useful postmortem analysis for doctor misses and no-car runs.
- Better-than-naive blackjack automation: split, surrender, insurance, peek, replay, and several item-aware choices are all considered.
- Good use of cumulative output files instead of generating many throwaway reports.

Most important next steps
1. Separate UI parsing from decision semantics.
2. Replace heuristic-only blackjack choices with EV-based evaluation.
3. Refactor quicktest into smaller policy modules with explicit reason tracing.
4. Turn doctor_missed, unlock conversion, and progression funnels into regression gates.
5. Remove or rename misleading tools like tools/test_gameplay.py.

Practical scorecard
- Reporting quality: strong
- Reproducibility: strong
- Completeness of game understanding in the current policy: weak to moderate
- Maintainability: weak to moderate
- True gameplay optimality: weak
- Regression-detection value: high

Final verdict
- Keep autotest.
- Trust it as a measurement harness.
- Do not trust the current autoplay to represent near-optimal play.
- The highest-value engineering investment is not more ad-hoc heuristics. It is a cleaner decision interface plus EV-driven policy evaluation.

Refined taxonomy takeaway
- The audit can now point to three distinct root-cause families with source grounding.
